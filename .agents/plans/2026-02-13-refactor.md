### 1B. Fix operator precedence in drift window calculation

**File:** `sync.rs:325`

```rust
// BEFORE — .max(1.0) binds to the literal 2.0, not the division result
end_window.min(60.0) / 2.0_f64.max(1.0)

// AFTER — explicit parentheses, dead .max(1.0) removed
(end_window.min(60.0) / 2.0).max(1.0)
```

The `.max(1.0)` on `2.0_f64` is dead code — `2.0.max(1.0)` is always `2.0`.
The likely intent was to clamp the *result* to at least 1 second. With the
default `end_window=120.0`, the output stays 30.0 — no behavior change for
typical inputs. But for small `end_window` values (e.g. 1.0), this now correctly
clamps to 1.0s instead of producing 0.5s.

Update the comment to be unambiguous:

```rust
// Use half the end window (capped at 60s, floored at 1s) for drift
// correlation — we already know approximately where the track should be.
```

**Verify:** `cargo test` — existing `test_drift_*` tests still pass.

### 1D. Fix confidence edge case for flat/degenerate correlations

**File:** `sync.rs:247-253`

```rust
// BEFORE — forces confidence=1.0 when second peak is ≤0, even if primary is also ≤0
let peak_ratio = if second_peak_value > 0.0 {
    peak_value / (second_peak_value + EPSILON)
} else {
    10.0
};

// AFTER — only claim strong match if primary peak is actually positive
let peak_ratio = if second_peak_value > 0.0 {
    peak_value / (second_peak_value + EPSILON)
} else if peak_value > 0.0 {
    // Primary peak is positive but there's no meaningful second peak.
    10.0
} else {
    // Both peaks are zero or negative — no meaningful correlation.
    1.0
};
```

With the original code, a flat/zero correlation array produces `confidence=1.0`.
The fix makes degenerate inputs produce `confidence=0.0` (since `(1.0-1.0)/0.5 = 0`).

### 1E. Use `f64::total_cmp` instead of `partial_cmp(...).unwrap()`

**File:** `sync.rs:231`

```rust
// BEFORE — panics on NaN
.max_by(|(_, a), (_, b)| a.partial_cmp(b).unwrap())

// AFTER — total order, NaN-safe
.max_by(|(_, a), (_, b)| a.total_cmp(b))
```

`f64::total_cmp` has been stable since Rust 1.62. It gives a total order where
NaN sorts high. The `power_to_db` amin floor of 1e-10 should prevent NaN in
practice, but this eliminates a latent panic for pathological inputs.

### 3B. Merge process/write loops to reduce peak memory

**File:** `main.rs:383-433`

**Depends on:** None.

Currently: loop 1 processes all tracks into `Vec<TrackResult>` (holding all
audio buffers), then loop 2 writes outputs. With N tracks, all audio lives in
memory simultaneously.

Refactor to: process each track, write its output immediately, drop the audio
buffer, keep only metadata for the summary/log.

Note: This will interleave per-track processing output with file writes. That's
fine — the goal is reducing peak memory.

```rust
// Sketch:
struct TrackSummary {
    path: PathBuf,
    offset: Option<f64>,
    confidence: Option<f64>,
    drift: Option<f64>,
    output_path: Option<PathBuf>,
    error: Option<String>,
}

let mut summaries: Vec<TrackSummary> = Vec::with_capacity(track_paths.len());

for track_path in track_paths {
    let result = process_track(&master_audio, master_sr, track_path, cli.sync_window);
    // write output immediately, extract summary, drop audio
    let summary = write_and_summarize(result, max_length, master_sr, &cli.output_suffix);
    summaries.push(summary);
}

// summary table + log use summaries (no audio buffers held)
```

This makes peak memory independent of track count: O(1) audio buffers instead
of O(N).

**Verify:** `cargo test`. Output WAVs, summary, and log should be unchanged;
CLI output ordering will change (interleaved processing/writing).

### 3D. Hoist per-frame allocations out of STFT loop

**File:** `mfcc.rs:102-123`

**Depends on:** None.

Three buffers are allocated per frame inside the STFT loop:

```rust
// BEFORE — inside the loop
let mut windowed_frame = fft.make_input_vec();   // ~16KB
let mut spectrum = fft.make_output_vec();         // ~16KB
let mut power = Vec::with_capacity(n_bins);       // ~8KB
```

For a 10-minute window (~51K frames), this is ~150K allocations. Move them
above the loop and reuse:

```rust
// AFTER — outside the loop
let mut windowed_frame = fft.make_input_vec();
let mut spectrum = fft.make_output_vec();
let mut power = vec![0.0f64; n_bins];

for frame_idx in 0..n_frames {
    let start = frame_idx * hop_length;

    for i in 0..n_fft {
        windowed_frame[i] = padded[start + i] * window[i];
    }

    fft.process(&mut windowed_frame, &mut spectrum).unwrap();

    for (k, complex_val) in spectrum.iter().enumerate().take(n_bins) {
        power[k] = complex_val.re * complex_val.re + complex_val.im * complex_val.im;
    }

    power_frames.push(power.clone());
}
```

Note: `power` is cloned into `power_frames` per frame. This preserves the
current API of `stft_power` returning `Vec<Vec<f64>>`. If 3E (streaming MFCC)
is implemented, this intermediate step becomes moot. But it's a clean standalone
improvement regardless.

**Verify:** `cargo test` — `test_extract_mfcc_shape` and
`test_mfcc_nonzero_for_rich_signal` confirm unchanged output.

### 3E. Stream MFCC pipeline to avoid materializing full STFT matrix (DEFERRED)

**Files:** `mfcc.rs:80-126` (stft_power), `mfcc.rs:338-385` (extract_mfcc)

**Depends on:** 3D (hoisted allocations make this cleaner).

Deferred until we have stronger MFCC regression tests. The current MFCC tests
primarily validate shape and non-zero output, which won't catch subtle numeric
differences introduced by floating-point reordering in a streaming refactor.

Currently `extract_mfcc` calls `stft_power` which returns the entire
`power_frames: Vec<Vec<f64>>` (~424MB for a 10-minute window). Then
`extract_mfcc` iterates over it one frame at a time.

Fuse the pipeline: compute each STFT frame, apply mel filterbank, convert to
dB, DCT, push coefficients — all in one pass per frame.

```rust
pub fn extract_mfcc(audio: &[f32], sr: u32, n_mfcc: usize, hop_length: usize) -> Vec<Vec<f64>> {
    let n_fft = DEFAULT_N_FFT;
    let n_mels = DEFAULT_N_MELS;
    let n_bins = n_fft / 2 + 1;
    let fmin = 0.0;
    let fmax = sr as f64 / 2.0;

    // Precompute reusable structures
    let window = create_hann_window(n_fft);
    let filterbank = create_mel_filterbank(sr, n_fft, n_mels, fmin, fmax);
    let mut planner = RealFftPlanner::<f64>::new();
    let fft = planner.plan_fft_forward(n_fft);

    // Center-pad audio
    let pad_length = n_fft / 2;
    let padded_len = pad_length + audio.len() + pad_length;
    // Convert f32 -> f64 inline per-frame instead of bulk allocation
    let n_frames = (padded_len - n_fft) / hop_length + 1;

    // Reusable per-frame buffers
    let mut windowed_frame = fft.make_input_vec();
    let mut spectrum = fft.make_output_vec();
    let mut power = vec![0.0f64; n_bins];
    let mut mel_energies = vec![0.0f64; n_mels];
    let mut mfcc_matrix: Vec<Vec<f64>> = vec![Vec::with_capacity(n_frames); n_mfcc];

    for frame_idx in 0..n_frames {
        let start = frame_idx * hop_length;

        // Window and convert f32 -> f64 per sample
        for i in 0..n_fft {
            let sample_idx = start + i;
            let sample = if sample_idx < pad_length || sample_idx >= pad_length + audio.len() {
                0.0
            } else {
                audio[sample_idx - pad_length] as f64
            };
            windowed_frame[i] = sample * window[i];
        }

        fft.process(&mut windowed_frame, &mut spectrum).unwrap();

        // Power spectrum
        for (k, c) in spectrum.iter().enumerate().take(n_bins) {
            power[k] = c.re * c.re + c.im * c.im;
        }

        // Mel filterbank
        for (m, filter) in filterbank.iter().enumerate() {
            mel_energies[m] = filter.iter().zip(power.iter()).map(|(&w, &p)| w * p).sum();
        }

        // dB + DCT
        let mel_db = power_to_db(&mel_energies, POWER_TO_DB_AMIN, POWER_TO_DB_TOP_DB);
        let coeffs = dct_ii_ortho(&mel_db, n_mfcc);

        for (idx, &val) in coeffs.iter().enumerate() {
            mfcc_matrix[idx].push(val);
        }
    }

    mfcc_matrix
}
```

This eliminates:
- The `padded` buffer (~212MB) — f32→f64 conversion happens per-sample inline
- The `power_frames` matrix (~424MB) — never materialized

`stft_power` becomes dead code after this change. Keep it if tests reference it
directly; otherwise remove.

**Verify (when un-deferred):** Add an MFCC numeric regression test (golden
values and/or tight tolerance) before refactoring, then run `cargo test`.

## Out of Scope

These were noted in the reviews but are not included in this plan:

- **`Async::<f32>` resampling** — Both reviews suggest switching rubato from
  `Async::<f64>` to `Async::<f32>` to eliminate f32→f64→f32 conversion buffers.
  This requires verifying rubato 1.0's `Async::<f32>` path works correctly with
  `audioadapter-buffers` and checking for precision regressions. Worth doing but
  needs its own investigation spike.

- **Parallel track processing (rayon)** — Both reviews note tracks could be
  processed in parallel. This interacts with memory (each parallel track holds
  its own buffers) and should wait until after 3B (process-then-write) and 3C
  (master MFCC caching) reduce the per-track memory footprint.

- **VAD `resample_for_vad` panic→Result conversion** — `vad.rs:346-365` uses
  `.expect()` where `audio.rs` uses `Result`. Low-impact since the panic paths
  are guarded by `WEBRTC_VALID_RATES`, but noted for future cleanup.

- **VAD unnecessary copy for compatible rates** — `vad.rs:86` does
  `audio.to_vec()` even when no resampling is needed. Could use `Cow<[f32]>`.

- **Mono downmix copy for mono input** — `audio.rs:226-228` copies the buffer
  via `.to_vec()` even when input is already mono.

- **Stale example output** — `assets/example-output.txt` doesn't match current
  CLI output. Should be regenerated from a real run.

- **Stale Python `.gitignore`** — `scripts/src/.gitignore` has Python/venv
  patterns. Low priority.
